Git Flow! — 概要と使い方

このページは Git Flow ブランチモデルの概要、代表的なコマンド、実際のワークフロー例を日本語でまとめたものです。主に git-flow 拡張（A successful Git branching model に基づく）を想定しています。

概要

Git Flow は、リリース管理に明確な役割を与えるブランチモデルです。主なブランチは次のとおりです：

  • main（または master）: 常にデプロイ可能な状態を保つ。本番リリースタグが付く。
  • develop: 次のリリースに向けた統合ブランチ。機能開発はここにマージされる。
  • feature/*: 新機能の開発用ブランチ。develop から分岐し、develop にマージされる。
  • release/*: リリース準備用ブランチ。バージョン番号の調整や最終テストを行い、完成したら main と develop にマージする。
  • hotfix/*: 本番での緊急修正。main から直接分岐し、修正後は main と develop にマージする。

メリットと注意点

  • メリット: 明確な役割分担によりリリース作業が整理される、複数の同時開発ラインを扱いやすい。
  • 注意点: ブランチ数が増えるため小規模チームや頻繁デプロイの環境（GitHub Flow 的な運用）には重い場合がある。

よく使うコマンド（git-flow 拡張）

事前に git-flow を導入している想定です（多くの環境で brew install git-flow、apt install git-flow 等）。

初期化

# リポジトリで一度だけ実行
git flow init

# 非対話でデフォルトを使う場合（例）
git flow init -d

Feature（機能）

# feature を開始
git flow feature start my-feature

# 作業後、develop にマージして feature 削除
git flow feature finish my-feature

Release（リリース準備）

# release を開始（例: 1.2.0）
git flow release start 1.2.0

# 最終調整後、リリース完了: main にマージしてタグ付け、develop にマージ
git flow release finish '1.2.0'

Hotfix（本番緊急修正）

# hotfix を開始（例: 1.2.1）
git flow hotfix start 1.2.1

# 修正後に完了
git flow hotfix finish '1.2.1'

ワークフロー例（短いフロー）

  1. 新機能は develop から feature/xxx を切って実装。
  2. 機能完成後、`git flow feature finish` で develop に統合。
  3. 一定の機能が揃ったら release/ を作成してテスト→タグ→ main へマージ。
  4. 緊急修正は hotfix/ を main から切って修正→ main/develop にマージ。

GitHub Flow と比較

GitHub Flow は main を中心に短い-lived ブランチ（通常は feature ブランチ）で素早くデプロイするモデルです。CI/CD が整備された環境やデプロイ頻度が高いプロジェクトでは GitHub Flow の方がシンプルで適しています。対して Git Flow はリリース作業が重視されるプロジェクトに向きます。

実践的な注意事項

  • チーム規模・リリース頻度によって採用可否を判断すること。
  • CI の設定（develop / release / main に対するビルド/テスト）を整備すること。
  • Pull Request の運用ルールを明確にし、マージフローを共有すること。
